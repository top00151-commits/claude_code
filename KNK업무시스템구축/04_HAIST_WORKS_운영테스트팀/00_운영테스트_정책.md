# 04 HAIST_WORKS 운영테스트팀 — 정책

> **역할**: 세션1·2·3 산출물을 **실제 직원이 쓰듯** 사용해보고, UX/업무적합성/입력부담/자료정확성 관점에서 문제를 끊임없이 발굴한다.
>
> **감사팀과의 차이**
> - 감사팀(00): 주장·수치의 **사실 여부**를 검증 (코드/DB/커밋 실측)
> - 운영테스트팀(04): 시스템을 **실제로 써보고** 업무 현실과 괴리되는 지점을 발굴 (UX·workflow·페인포인트)
> 감사팀이 "뼈대가 정확한가"를 본다면, 운영테스트팀은 "살이 붙어서 실제로 쓸 만한가"를 본다.

## 절대 원칙

> **2026-04-28 강화**: `00_검증깊이_강화정책_v2.md` 발행. 5라운드부터 Level-1(도달성) → Level-2(입력 검증) → Level-3(업무 완주) → Level-4(엣지·부정) 4단계 모델 적용.

1. **추상 금지** — "뭐뭐가 불편할 것 같다"는 금지. 반드시 실제 URL을 띄우고 클릭·입력해보고 기록한다.
2. **페르소나 이탈 금지** — 테스트 중에는 지정된 페르소나의 지식·권한·업무 루틴을 고수한다.
   - 신입은 용어를 모르고, 베트남 직원은 한국어가 불편하며, 팀장은 10분 안에 결재해야 한다.
3. **시나리오 완주** — 한 페르소나가 오늘 해야 할 업무를 **끝까지** 시뮬레이션한다. 한 페이지만 찍고 끝내지 않는다.
4. **세션 경계 준수** — 문제 발견 시 01/02 세션에 피드백만 남기고, **코드를 직접 수정하지 않는다** (감사팀과 동일 원칙).
5. **"일이 늘어나는가" 질문** — 모든 테스트 끝에 "이 시스템을 쓰면 원래 업무보다 일이 늘어나는지 줄어드는지"를 반드시 판정한다.
6. **⛔ 비간섭(Non-Interference)** (2026-04-21 대표 추가, 절대준수) —
   - **다른 세션이 작업 중일 때는 서버 기동·DB 접근·파일 쓰기 테스트 금지**. 파일 읽기·문서 작성만 허용.
   - uvicorn의 `--reload` 는 다른 세션이 main.py·templates·database.py 를 수정하면 자동 재시작하여 충돌을 유발한다. 따라서 기동은 **반드시 "다른 팀 작업 중지" 확인 후**.
   - 기동 전 체크리스트:
     1. 09 팀장 `01_세션상태보드.md` 의 01 메인 상태가 "대기" 혹은 "회신 완료"인가?
     2. `git status` 에서 `01_HAIST_WORKS/` 관련 미스테이지 변경이 없는가?
     3. 대표 또는 팀장이 "운영테스트 시작해도 됨" 신호를 보냈는가?
   - 위 3개 중 하나라도 불만족 시 **파일 기반 검토(코드 읽기·과거 로그 분석)만** 수행.
   - 기동한 뒤에도 **세션당 최대 10분 이내**로 끝내고 서버 프로세스 종료 (`taskkill`).
