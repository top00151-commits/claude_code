## C. 시각 시나리오 (도크 열림 / 닫힘)

### C-1. base_sales.html — 도크 닫힘 (body.dock-open 없음)
- `.layout-wrap` margin-right: 0 → 전체 가로폭 차지
- `.main-wrap` margin-right: 0 → flex:1 로 layout-wrap 안에서 사이드바(264px) 제외 나머지 차지
- `<main class="main has-sidebar logi-main">` 적용 규칙:
  1. `.main` (line 140) padding:0 24px
  2. `.main.has-sidebar` (line 642) padding:14px 20px
  3. `.logi-main` (line 3084) padding:24px 32px 48px, width:100%, max-width:none
  4. `.main` (line 1376, !important) padding:24px 28px !important
  5. `.logi-body .main.has-sidebar.logi-main` (line 3358) **미적용** (body=sales-body)
- **최종 padding**: `!important` 인 line 1376 가 우세 → **padding: 24px 28px**
- **여백 원인 시 도크 닫힘**: 우측 끝 padding 28px (정상). 추가 여백 없음 ✓

### C-2. base_sales.html — 도크 열림 (body.dock-open)
- `.layout-wrap` margin-right: **420px** (인라인 line 230) — 전체 layout 이 420px 좌측 시프트
- `.main-wrap` margin-right: **420px** (style.css line 3638) — main-wrap 도 추가 시프트?
  - **하지만** main-wrap 은 layout-wrap 의 자식. **margin-right 는 부모 컨테이너 내부 우측 공간으로 작동한다.**
  - `.layout-wrap` 가 `display:flex; overflow:hidden`. **자식의 margin-right 가 부모 안쪽에서 우측 공간을 만들면, 그만큼 자식이 좁아진다.**
  - 즉 layout-wrap 이 우측을 420px 비워둔 상태에서 그 안의 main-wrap 도 추가 420px 우측 비움 → **최대 420 + 420 = 840px 좌측 시프트 + 우측 빈 공간**
- **여백 원인 시 도크 열림**: ⚠ **이중 시프트 — 컨텐츠가 최대 840px 좁아질 가능성**

### C-3. base.html — 비교 (도크 열림)
- `.layout-wrap` margin-right: 0 (인라인 override 없음)
- `.main-wrap` margin-right: 420px (G5)
- `.main.has-sidebar` margin-right: 0 (line 3639, 이중 방지)
- 결과: 사이드바 고정, 컨텐츠만 420px 좌측 시프트 ✓ 의도대로 동작
- **base.html 은 정상, base_sales/logi 는 이중 시프트 발생**

### C-4. flex layout 의 실제 동작 — 정밀
`.layout-wrap` 은 `display:flex` (line 637). 자식: `.sidebar` (width 260px, flex-shrink:0) + `.main-wrap` (flex:1).
- 도크 열림 시:
  - layout-wrap 자체의 margin-right 420px → 부모 body 의 가용폭에서 우측 420px 비움
  - layout-wrap 내부 가용폭 = (viewport - 420) 그대로 사이드바(260) + main-wrap(나머지)
  - **다시** main-wrap 자식이 margin-right:420 → main-wrap 의 시각 폭이 420px 더 좁아짐 → 우측에 ❌ **빈 공간 420px** 발생
- **이게 "빅터AI 사이드바와 메인창 사이 여백 큼" 의 정체다.**

### C-5. 사이드바와 도크 사이 시각 거리 (도크 열림 시)
- viewport 1920 가정
- layout-wrap 시작: x=0, 폭 1500 (1920-420)
- 사이드바 x=0~260
- main-wrap x=260, 가용폭 1240
- main-wrap 안 main 폭 = 1240 - margin-right(420) = **820** (실제 main 폭)
- main 우측 끝 = 260+820 = 1080
- 도크 시작 x = 1500 (도크 fixed 우측 0, 폭 420 → x=1500~1920)
- **main 우측 끝(1080) ↔ 도크 시작(1500) 간격 = 420px** ❌
- **이 420px 가 대표가 "여백이 많다" 라고 한 그것**

### C-6. base_sales 도크 닫힘 시 컨텐츠 영역
- viewport 1920, layout-wrap margin-right 0, main-wrap margin-right 0
- main 폭 = 1920 - 260 = 1660. 우측 끝 = 1920. 정상 (여백 없음)

## D. 가능한 원인 후보 4건 검증

### D-1. logi-main 잔존 — base_sales 가 base_logi 복사하면서 그대로
**검증**: base_sales.html line 115 = `<main class="main has-sidebar logi-main">`. logi-main 클래스 자체는 `.logi-body` 가 아닌 한 광역 selector (line 3084) 만 적용 → padding 24px 32px 48px. **여백의 직접 원인 아님** (max-width:none, width:100% 이라 폭 제한 없음).
그러나 `.logi-body .main.has-sidebar.logi-main` (line 3358) 의 padding 20px 28px 가 sales-body 에는 미적용 → 약간 패딩 차이 있으나 우측 420px 빈공간의 원인 아님.
**판정**: 부차적, 여백 본질은 아니다.

### D-2. dock-open 핫패치 미적용
**검증**: body 클래스가 sales-body / logi-body 여도 `body.dock-open` 셀렉터는 정상 매칭(`body` + `.dock-open`). G5 (line 3637-3638) 적용됨 + 인라인 layout-wrap 시프트도 적용됨.
**판정**: 미적용이 아니라 **두 개 모두 적용되어 이중 시프트** 가 본질.

### D-3. main-wrap dock-rail spacer 부재
**검증**: spacer 컴포넌트 없음. flex margin-right 로만 시프트.
**판정**: spacer 부재 자체는 의도된 디자인. 문제 아님.

### D-4. 3 base 의 main 구조 차이 — ⚠ **핵심 원인**
**검증**:
- base.html: `.main-wrap` 만 시프트 (G5 단일)
- base_sales/logi: **`.layout-wrap` 시프트 (인라인) + `.main-wrap` 시프트 (G5) 이중 적용**
- 두 시프트는 부모-자식 관계 (layout-wrap > main-wrap) 라 시각상 누적되어 컨텐츠 우측에 420px 빈공간을 만든다.
**판정**: ✅ **본질적 여백 원인 — 이중 margin-right shift**

### D-5. 추가 — 페이지 헤더 시프트 비교
- base.html: top_fixed block 은 `.main-wrap` 안 → main-wrap 시프트로 함께 좌측 이동
- base_sales/logi: top_fixed block 없음. layout-wrap 시프트로 사이드바·메인 모두 좌측 이동
- 즉 sales/logi 는 **사이드바도 같이 끌려간다**. 이는 유저가 "사이드바와 도크 사이 거리가 변한다" 라고 느낄 수 있는 원인.
