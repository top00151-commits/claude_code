## 1. 6대 동적 변수 정의

사용자 제시 변수를 명확하게 분류:

| # | 변수 | 예시 | 영향 영역 |
|---|---|---|---|
| ① | **사양 변경 (major)** | 부품 ABC → DEF 교체 | BOM · 원가 · 제조 |
| ② | **시간별 단가 변경** | 원자재 시세 월별 변동 | 원가 · 마진 |
| ③ | **환율 변경** | USD/KRW, VND/KRW 변동 | 수입·수출 원가 |
| ④ | **수량별 단가** | 100개 구매 vs 1000개 구매 단가 차이 | 구매 타이밍·묶음 |
| ⑤ | **납기 변동** | 공급사 리드타임 2주 → 6주 | 생산 일정·재고 |
| ⑥ | **조금 사양 변경 (minor)** | 커넥터 위치·나사 규격 미세 조정 | Variant BOM · 재고 분리 |

### 1.1 KNK 업종 특성상 모든 변수가 동시 발생

**장비 제조업(ETO·MTO) 특징**:
- 수주별로 다른 사양 (변수 ①·⑥ 일상 발생)
- 장비당 수백~수천 부품 (변수 ②~⑤ 부품별 차등 발생)
- 본사 + 베트남 법인 이중 환율 (변수 ③ 필수)
- 고객사별 발주 수량 차이 (변수 ④)

→ **단일 변수만 잘 처리해도 부족** — 6개 동시 관리 가능한 시스템 필요

## 2. 변수 ①: 사양 변경 (ECO)

**출처**: Acumatica, SolidWorks DELMIAWorks, Arena, TGI ERP, Sibe.io

### 2.1 ECO (Engineering Change Order) 시스템이 제공하는 기능 (팩트)

| 기능 | 내용 |
|---|---|
| **영향 분석** | 변경이 영향 주는 부품·어셈블리·문서 자동 리스트 |
| **비용 비교** | **변경 전/후 BOM 비용 자동 비교** (costing summary) |
| **리비전 관리** | BOM 버전 이력 + 특정 시점으로 롤백 가능 |
| **Effectivity Date** | 언제부터 새 BOM 적용? 재고 소진 후 vs 즉시 |
| **MRP 재계산** | 변경 승인 시 MRP 자동 재실행 (요구량·리드타임·계획 오더 갱신) |

### 2.2 실사용 경고 (팩트)

> **"BOM 버전이 바뀌면 MRP가 요구량·리드타임·계획 오더를 재계산하는데, 자재 계획이 cutover 시점에 동기화되지 않으면 **구식 부품을 구매하거나 새 부품을 누락**하는 사고 발생"**

— Oxmaint, Influential Women

### 2.3 2024~2026 AI 기반 새로운 기능 (팩트)

**자연어 모델 (NLP)이 BOM 계층 + CAD 메타데이터 분석** → 사양 변경 시 어느 다운스트림 어셈블리가 깨질지 예측
**파일럿 결과**: **엔지니어링 평가 시간 83% 감소**

(출처: Influential Women 2026)

### 2.4 주요 시스템 기능 비교

| 시스템 | ECO 특화 |
|---|---|
| **Acumatica Cloud ERP** | Engineering Change Control 모듈 |
| **SolidWorks DELMIAWorks** | ECO + Quality 모듈 결합 |
| **Arena PLM** | 의료기기 FDA 규제 수준 ECO 추적 |
| **TGI ERP** | Web-based ECO management |

## 3. 변수 ②: 시간별 단가 변경

**출처**: 한국 ERP 자재관리 공식 구성, Microsoft D365, Rockton, OpenBOM

### 3.1 한국 ERP의 단가 관리 공식 구조 (팩트)

> **"매입처 정보·기본 단가 등록 → 사업장·공장별로 **적용일자·협의일자·통화·확정단가/가단가/견적단가** 여부 세부 정보 입력 → **관리자 승인**으로 확정"**

> **"구매 단가 변동 시 **변경일자·단가협의일·적용시작일**을 별도 입력 관리 + 관리자 승인만 처리"**

(출처: Decomsoft ERP, emaxit Frame7)

### 3.2 핵심 기능: Date-Sensitive Pricing

**Microsoft D365 Business Central 공식 기능**:
- 고객별 단가
- 수량 구간 단가 (quantity-break)
- **날짜 기반 단가** (date-sensitive pricing)
- 관리자 승인 후 적용

**OpenBOM 기능**:
- **리비전 이력이 비용 변화 추적**
- "지금 비용 얼마 + 미래 비용 얼마" 조회 가능
- 실시간 BOM 코스팅으로 가격 변동성 대응

### 3.3 단가 구분 (3종) — 한국 ERP 표준

| 구분 | 의미 |
|---|---|
| **확정단가** | 공급사와 협의 완료된 최종 단가 |
| **가단가** | 협의 전 임시 단가 (프로젝트 견적용) |
| **견적단가** | 공급사 견적 단계의 단가 (계약 전) |

### 3.4 관리자 승인 워크플로우 (팩트)

모든 단가 변경 시:
1. 담당자가 변경 등록 (변경일자·협의일·적용시작일)
2. 근거 자료 첨부
3. 관리자 승인 대기
4. 승인 후 확정단가로 전환
5. **이력 영구 보존** — 과거 단가 조회 가능

## 5. 변수 ④: 수량별 단가 (Tiered Pricing)

**출처**: Wantek, V Bundles, SAP S/4HANA, MRPeasy, ERPAG, Rockton, DealHub

### 5.1 수량별 단가 2대 모델 (공식)

#### 5.1.1 All-Units Discount (전량 할인)

**정의**: 총 주문량이 티어 임계값 도달 시, **전체 수량에 할인 단가 적용**

**예시**:
- 1~99개: 개당 10,000원
- 100~499개: 개당 9,000원
- 500개~: 개당 8,000원

→ 500개 주문 시 전체 500개가 개당 8,000원

#### 5.1.2 Incremental (Marginal) Discount

**정의**: 티어 내 수량만 해당 할인 적용

**예시** (위 동일 티어 기준):
- 500개 주문 시:
  - 1~99개 = 99개 × 10,000원
  - 100~499개 = 400개 × 9,000원
  - 500개 = 1개 × 8,000원

### 5.2 ERP 지원 현황 (2026)

| ERP | 기능 |
|---|---|
| **SAP S/4HANA** | Volume discount pricing 표준 지원 |
| **Microsoft D365** | customer-specific + quantity-break + **date-sensitive** 결합 |
| **MRPeasy** | Tiered Pricing 데모 공식 (비디오 가이드) |
| **ERPAG** | Tier-pricing 공식 기능 |
| **Rockton Pricing Management** | Dynamics 365 확장 프라이싱 |
| **Salesforce CPQ** | Volume-based pricing 표준 |

### 5.3 실제 효과 (팩트)

Shopify 판매자 기준:
- Quantity breaks 도입 후 **평균 주문가 (AOV) 68% 증가** (소비재·첫 달 내)

**제조 조달 관점**:
- 공급사별 수량 구간 협상
- **자동 발주 시 최적 수량 계산** (안전재고 + 수량 할인 고려)

## 6. 변수 ⑤: 납기 변동

**출처**: IFS Cloud, SAP S/4HANA Predictive MRP, Siemens MRP II, Sotatek

### 6.1 기존 MRP의 납기 처리 한계

**문제점**:
- 공급사 리드타임이 협약 리드타임과 실제 리드타임 불일치

- 기존 MRP는 **배치 처리**라 실시간 반영 지연

### 6.2 2026 실시간 MRP (S/4HANA)

**공식 기능**:
- **In-memory computing**으로 실시간 MRP 처리
- 배치 처리 지연 제거
- 자재 요구사항 실시간 재계산

### 6.3 Material Planning Simulation (IFS Cloud)

**공식 기능**:
- 다양한 수요·공급 시나리오 모델링
- **라이브 생산 데이터에 영향 없이** 평가
- 공급망·제조 운영에 끼치는 영향 분석
- **병렬 시나리오 실행** 가능

### 6.4 납기 변동 대응 패턴 (업계 공통)

| 기능 | 내용 |
|---|---|
| 공급사별 리드타임 이력 DB | 과거 3개월·6개월·1년 평균 자동 계산 |

| 백오더 관리 | 납기 초과 발주는 고객 주문 영향 추적 |

## 7. 변수 ⑥: 조금 사양 변경 (Variant BOM)

**출처**: SAP Variant Configuration, D365, Epicor, Infor CloudSuite Industrial

### 7.1 Variant BOM이란 (팩트)

**정의**: 기본 BOM 위에 **옵션·변형**을 파라미터로 관리

**예시**:
- 기본 장비 모델 X
- 옵션 A: 커넥터 타입 1 or 타입 2
- 옵션 B: 볼트 규격 M6 or M8
- 옵션 C: 전원 110V or 220V

→ 기본 BOM 1개 + 옵션 파라미터로 **수백 개 변형 관리 가능**

### 7.2 SAP Variant Configuration (SAP VC)

**공식 기능**:
- 파라미터 기반 BOM 생성
- 고객 주문 시 옵션 선택 → 자동 BOM 구성
- CPQ (Configure-Price-Quote) 연동

### 7.3 장비 제조업 특화 (ETO)

**Epicor ERP** (ETO 설비산업 특화):
- **수주형태별 프로세스**: 신규·개조·Spare·데모·C/S·상품
- 프로젝트별 BOM 분기 관리
- 각 수주마다 다른 BOM을 기본 BOM의 변형으로 관리

### 7.4 Variant BOM vs 일반 BOM 장단점

| 항목 | Variant BOM | 일반 BOM |
|---|---|---|
| 초기 설계 | 복잡·시간 많이 소요 | 간단 |
| 변형 추가 | 옵션만 추가 → 쉬움 | 새 BOM 복사 필요 |
| 재고 정합성 | 옵션 공통 부품은 공유 | 독립 재고 |
| 사양 변경 | 옵션 하나만 수정 | 모든 변형 수동 수정 |
| KNK 장비 수주별 차이 | 적합 | 부적합 |

| ⑥ Variant BOM | 옵션 파라미터 기반 + 수주형태별 분기 | SAP VC · Epicor |

### 12.2 2026 AI 접목 기회

| **ECO 영향 분석** | GPT 수준 BOM 계층 분석 → 다운스트림 어셈블리 영향 예측 (83% 시간 절감 사례) |

### 12.3 피할 함정 (업계 공통 실수)

| # | 함정 | 원인 |
|---|---|---|
| 1 | **BOM cutover 시 구식 부품 구매·신규 부품 누락** | 자재 계획과 BOM 변경 동기화 실패 |
| 2 | 단가 이력 없이 최신 단가만 저장 | 원가 분석·소송 대응 불가 |
| 3 | 환율 수동 입력 | 오타·지연·일관성 없음 |
| 4 | 수량 할인을 공급사마다 수동 계산 | 누락·오류 빈발 |
| 5 | 납기 변동을 엑셀로만 추적 | 생산 일정 자동 반영 불가 |

| 7 | AI 조달을 맹신 | Nike 재고 예측 실패 교훈 — Human-in-the-loop 필수 |

### 12.4 HAIST_WORKS 자체 구축의 장점 (KNK 맥락)

**6대 변수 대응을 자체 설계 시**:
- 관리코드 8자리 (`001T2604`) 그대로 유지 가능
- 12부서 설문 응답 기반 **KNK 특화 필드** 추가
- Claude API로 AI 예측 조달을 **SaaS 구독비 없이 API 사용량만**
- 베트남 법인 VND·KRW 동시 처리 설계 가능

### 12.5 업계 공개 사례 한계 (재확인)

- 구체 자재관리 동적 변수 처리 노하우는 **대부분 비공개**
- 유사 기업 원익IPS·주성엔지니어링·테크윙 등 구체 기능 확인 불가
- **공식 ERP 매뉴얼 · 학술 논문 · AI 벤더 공식 기능 문서만** 신뢰 가능한 정보원
