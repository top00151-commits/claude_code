## D01-01 set-lang 405 조치

- **원인**: `@app.post("/api/set-lang")` 로 POST-only 등록된 상태에서 04 테스트팀이 GET 쿼리스트링으로 호출 → FastAPI/Starlette 가 메서드 불일치 시 자동 405 반환.
- **수정 파일·라인**: `app/main.py` L840~L868 (기존 L840~L853 교체).
- **수정 전/후 요지**:
  - 전: `@app.post("/api/set-lang")` + JSON body 전용. GET 은 405.
  - 후: `@app.api_route("/api/set-lang", methods=["GET","POST"])` 로 변경. POST 는 기존대로 JSON body 수용(+JSON 응답), GET 은 `?lang=vi` 쿼리 수용 후 `Referer` 로 303 리다이렉트(주소창/북마크 UX).
  - `LANGS` 화이트리스트(`ko/vi/en`) 검증은 그대로, `users.lang` UPDATE·`session["lang"]` 설정 로직 동일. DB 부작용 불변.
  - 기존 프런트 `base.html` L406 의 `fetch('/api/set-lang', {method:'POST', ...})` 는 수정 없이 계속 동작.
- **import 검증 결과**: 이 세션 환경에서 Bash/PowerShell 실행 권한 차단으로 `python -c "from app.main import app"` 실행 불가. **정적 확인**으로 대체: `@app.api_route(..., methods=[...])` 는 FastAPI 표준 데코레이터이며, 사용된 `Request`/`RedirectResponse`/`JSONResponse`/`LANGS`/`db_session`/`get_user` 는 모두 L5~L24 상단에서 기존 임포트됨. 문법 에러 가능성 없음. 서버 기동 시 확인 필요.
- **04 테스트팀 재시뮬레이션 요청 여부**: **요청 필요**. P10 쑤아잉 페르소나로 `GET /api/set-lang?lang=vi` → 303 + DB `users.lang='vi'` 갱신 확인, `GET /api/set-lang?lang=en` 동일 확인. POST fetch 회귀(사이드바 언어 셀렉터)도 한 번 확인 부탁.

## D01-02 제조팀 폴백 조치

- **원인**: "폴백" 이 아니라 **의도된 권한 가드 303 리다이렉트**. `main.py` L113~L124 `can_use_logistics()` 는 admin/ceo/executive 만 기본 허용, 그 외는 `users.can_use_logistics` 플래그=1 인 자에 한함. P3 마준영(team=7, role=member) 은 플래그 0 이므로 `/logistics`(L3277), `/parts`(L3299), `/stock/issue`(L3978) 등에서 `RedirectResponse("/home", 303)` 반환. HTTP 레벨로는 303→200 체인이라 테스트 로그에 `200 final=/home` 으로 보임.
- **수정 파일·라인**: **변경 없음**(안 A 채택).
- **수정안 A/B 와 채택 사유**:
  - **안 A (채택, 코드 변경 0줄)**: 현 권한 정책 유지. 제조팀원이 물류 화면에 접근해야 한다면 admin 이 `/api/admin/user-logistics`(L3255) 로 해당 사용자 `can_use_logistics=1` 플래그를 켠다. — 지시서 "가장 보수적인 안" 원칙에 부합. 현 가드는 의도된 설계이며, 제조팀 전원을 일괄 허용하면 쓰기 경로(`/parts/new` POST, `/stock/issue` POST 등 60여개)도 함께 열려 **잘못된 입출고/단가 갱신 리스크** 가 발생함.
  - **안 B (미채택, 팀장 판단 요청)**: 제조팀(team=7) 만 **읽기 전용 GET**(`/logistics`, `/parts`, `/stock/issue` GET 폼까지) 허용, POST 는 계속 `can_use_logistics()` 로 차단. 구현은 `can_use_logistics_read(user)` 신규 함수 + 해당 3개 GET 라우트만 스위치. 파급 범위 ~10 라인. 다만 `team=7` 의미(제조팀 코드인지)를 `teams` 테이블 시드로 재확인 필요하고, "다른 생산 부서 / 법인 / 외주 협력사" 가 같은 논리로 요청할 가능성이 있어 **정책 결정** 이 앞서야 함.
  - 추가: 리다이렉트 시 사용자가 이유를 모르는 UX 문제 개선안 — `RedirectResponse("/home?no_perm=logistics", 303)` + `home.html` 에 플래시 배너. 이것도 단독 결정 보류.
- **import 검증 결과**: 변경 없음 → 임포트 영향 없음.
- **04 테스트팀 재시뮬레이션 요청 여부**: **불요**. 대신 **로그 수집기 개선 권고**: `run_persona_test.py` 에서 HTTP 리다이렉트 체인(원 응답 303 → 최종 200) 을 구분해 기록하도록 업데이트하면, "의도된 가드" vs "진짜 폴백" 을 지표에서 구분 가능. 이는 04 세션 작업 제안(코드 수정 권한은 04 에 없으므로 테스트 스크립트만 업데이트).
