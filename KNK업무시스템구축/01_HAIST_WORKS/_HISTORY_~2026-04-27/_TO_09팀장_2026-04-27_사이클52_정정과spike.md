## 작업 B — S3-1 stock_balances VIEW DDL 결함 spike 수정

### B-1. 결함 원인 (04 운영검증 발견 사실 확정)
- `app/database.py` **line 221**: `parts.part_name TEXT NOT NULL` ← 실제 parts 컬럼명은 `part_name`
- `app/database.py` **line 743 (수정 전)**: `p.name AS part_name` ← parts 테이블에 `name` 컬럼 없음 → SQL error
- `app/database.py` **line 749 (수정 전)**: `GROUP BY p.id, p.part_no, p.name, p.unit` ← 동일 결함

**우회 사유**: 헬퍼들(line 3630, 3704, 3755 등)은 VIEW의 `part_name` 컬럼을 쓰지 않고 `parts p` 직접 JOIN 후 `p.part_name` 사용 → VIEW 자체가 막상 생성되지 않아도 헬퍼는 동작. 하지만 VIEW가 한 번이라도 CREATE 되면 `no such column: p.name` 에러 → S2 4차 +S3-1 운영 위험.

### B-2. 수정 내용 (database.py line 738~752)
```sql
-- 수정 후
DROP VIEW IF EXISTS stock_balances;
CREATE VIEW stock_balances AS
SELECT  p.id                                  AS part_id,
        p.part_no                             AS part_no,
        p.part_name                           AS part_name,    -- p.name → p.part_name 정정
        COALESCE(SUM(sm.quantity), 0)         AS on_hand,
        p.unit                                AS unit,
        MAX(sm.occurred_at)                   AS last_movement_at
FROM    parts p
LEFT JOIN stock_movements sm ON sm.part_id = p.id
GROUP BY p.id, p.part_no, p.part_name, p.unit;     -- p.name → p.part_name 정정
```

### B-3. 마이그레이션 안전성
- 기존 `CREATE VIEW IF NOT EXISTS` → **`DROP VIEW IF EXISTS` + `CREATE VIEW`** 패턴 채택
- 결과: 운영 중 DB가 (1) VIEW 미존재 → 정상 생성, (2) 결함 VIEW 존재 → 자동 교체, (3) 정상 VIEW 존재 → 자동 재생성 (idempotent 유지)
- VIEW의 외부 노출 컬럼명은 `part_name` 그대로 → 모든 헬퍼/라우트 측 수정 불필요 (영향 0건)
