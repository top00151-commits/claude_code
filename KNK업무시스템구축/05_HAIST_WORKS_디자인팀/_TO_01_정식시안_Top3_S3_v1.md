## 1. 3계층 위임 화면 ASCII (CEO → 팀장 → 팀원)

### 1-A. CEO 화면 — "물류 권한"을 팀장에게 위임
```
+--------------------------------------------------------------+
| 새 위임                                              [닫기]  |
+--------------------------------------------------------------+
| (1) 누구에게      [ 박팀장 ▾ ]   ← 사용자 검색              |
| (2) 무엇을        [✓] 자재 read   [✓] 자재 write             |
|                   [ ] 자재 approve  [ ] 매출 read            |
|                   ※ 본인 보유 권한만 노출 (No escalation)   |
| (3) 언제까지      ◉ 30일  ○ 90일  ○ 직접입력 [____]          |
|                   ○ 영구 (관리자 승인 필요)                  |
| (4) 사유          [_____________________] (감사로그용, 선택) |
| (5) 재위임 가능   [○ ON] [● OFF] ← OFF가 기본값             |
+--------------------------------------------------------------+
|                                  [ 취소 ]   [ 위임 발송 ]    |
+--------------------------------------------------------------+
```

R3 §4-2의 4필드(누구/무엇/언제/사유) + §4-5의 재위임 토글(OFF 기본).

### 1-B. 팀장 화면 — 받은 권한을 팀원에게 재위임
```
+--------------------------------------------------------------+
| 박팀장 (자재팀장) — 받은 권한 + 재위임                       |
+--------------------------------------------------------------+
| [받은 권한]                                                  |
|  · 자재 read    (CEO → 박팀장 / 만료 5/25)                  |
|  · 자재 write   (CEO → 박팀장 / 만료 5/25 / 재위임 OFF)     |
+--------------------------------------------------------------+
| [재위임 시도] → 재위임 OFF 항목은 "위임 불가" 안내           |
+--------------------------------------------------------------+
```

### 1-C. 팀원 화면 — 내가 가진 권한 조회
```
+--------------------------------------------------------------+
| 김사원 — 내 권한                                              |
+--------------------------------------------------------------+
| 직접 부여(역할):  MEMBER (자재 read)                         |
| 그룹 상속:        자재팀 그룹 → 자재 read                    |
| 위임 받음:        박팀장 → 입고처리(write) / 만료 5/23       |
| ────────────────────────────────────────────────────────────  |
| 합계 권한:  자재 read ✓ / 자재 write ✓(한시) / approve ✗     |
+--------------------------------------------------------------+
```
R3 §3 의사코드 인용 — `collectAllPerms(user)` 결과를 3줄로 시각화.

## 2. Group Inheritance + Delegation Token 패턴 시각화

```
[CEO 김정락 OWNER]
   │
   │ Inheritance (③) — MANAGER 그룹은 자재 read/write 자동 보유
   ▼
[자재팀장 박OO MANAGER 그룹]
   │
   │ Delegation Token (④) — "입고처리(write)" 만 한시 부여
   │   expires_at = 2026-05-23
   │   재위임 OFF
   ▼
[김사원 MEMBER + 한시토큰]
```

UI 표현:
- 트리 노드 옆 **태그**: `[OWNER]` `[MANAGER]` `[MEMBER]` `[VIEWER]` (R3 §6 — 4역할).
- 위임받은 권한은 **점선 박스 + ⏱ 아이콘**으로 한시성 시각화.
- 회수 시 **Cascade 화살표** 점선으로 영향 표시.

## 3. 만료일·재위임OFF·감사로그 3원칙 UI

R3 §1의 3원칙 (a) 위임자 권한 범위 내 / (b) 만료일 / (c) immutable 로그.

### 3-1. 위임 카드 상단 3 배지
```
+----------------------------+
| 박팀장 — 자재 write        |
| [⏱ D-29]  [🔒 재위임OFF] [📜 감사로그]  ← 3 배지 강제 노출  |
| 부여 2026-04-25 / 만료 5/25                                  |
| 사유: "출장 대체 결재"                                       |
| [ 회수 ]                                                     |
+----------------------------+
```
3 배지는 **모든 위임 카드에 강제 노출** — 빠뜨릴 수 없는 안전장치.

### 3-2. 만료 임박 표시
- D-7 이내: `var(--st-progress)` 노란 테두리
- D-1 이내: `var(--st-delay)` 빨간 점멸 (CSS 애니메이션, 1Hz)
- 만료됨: `tag-wait` 회색 + "EXPIRED" 라벨

## 4. Deny overrides Allow 충돌 표시

R3 §2 인용 — "Deny overrides Allow". 백엔드 미들웨어에서 강제하지만 UI 안내도 명시.

```
[권한 부족 토스트]
┌────────────────────────────────────┐
│ ⚠ 이 작업은 '매출 approve' 권한이  │
│   필요합니다.                      │
│   현재 상태: 자재팀 그룹에 DENY     │
│   적용 (회계팀장 명시 차단)        │
│   [ 팀장에게 요청 ]   [ 닫기 ]     │
└────────────────────────────────────┘
```

R3 §4-3 — "팀장에게 요청" 버튼 클릭 시 위임 요청 워크플로 모달 오픈.

권한 평가 표시 (디버그 패널, OWNER만):
```
계산 순서:  DENY 확인 → ALLOW 확인 → 기본 거부
사용자: 김사원
요청 자원: 매출 approve
  · 직접권한:  ✗ 없음
  · 그룹상속:  자재팀 → 매출 권한 없음
  · 위임토큰:  없음
  · DENY 항목: 회계팀장 → "비회계자 매출 approve DENY"
결과: ✗ 거부 (DENY overrides)
```

## 5. 페르소나 관점

### CEO 위임 (P1 — 김정락)
- 좌 트리에서 팀장 노드 우클릭 → "권한 위임" 모달.
- 본인 OWNER 권한 전체 풀에서 체크박스 선택, 만료 30일 기본.

### 팀장 위임 (P2 — 박팀장)
- 받은 권한 중 **재위임 ON** 항목만 팀원에게 재위임 가능.
- 재위임은 **만료일이 본인 만료일 이내**로 강제 (기간 초과 입력 시 토스트 차단).

### 팀원 권한 조회 (P3 — 김사원)
- "내 권한" 화면에서 직접/상속/위임 3 출처 분리 표시.
- 권한 부족으로 차단된 액션은 회색 + 자물쇠 아이콘 + 호버 시 사유 툴팁.

## 6. 회수·이력 조회 화면

### 6-1. 회수 (R3 §4-6 — 빨간색 + 2단계 확인 + Cascade 미리보기)
```
+--------------------------------------------------------------+
| ⚠ 회수 확인                                          [닫기] |
+--------------------------------------------------------------+
| 다음 권한을 회수합니다:                                       |
|   · 박팀장 — 자재 write                                       |
|                                                               |
| 영향 범위 (Cascade):                                         |
|   · 김사원에게 재위임된 '입고처리' 권한 1건도 함께 무효화    |
|   · 진행 중인 검수 GR-9981 작업이 권한 없음 상태로 전환      |
|                                                               |
| 본 작업은 immutable 로그에 기록되며 되돌릴 수 없습니다.      |
+--------------------------------------------------------------+
| [확인 1단계] "회수합니다" 입력 [____]                         |
|                                  [ 취소 ]  [ 회수 실행 ]      |
+--------------------------------------------------------------+
```
회수 버튼 색상: `var(--st-delay)` 배경 + 흰 텍스트.

### 6-2. 이력 조회 (R3 §4-4 — 시간역순 타임라인)
```
[이력]
2026-04-25 14:30  김정락 → 박팀장   GRANT  자재 write   만료 5/25
2026-04-25 09:12  박팀장 → 김사원   DELEGATE 입고처리   만료 5/23
2026-04-24 17:50  김정락 → 정팀장   REVOKE  매출 approve
2026-04-23 11:05  김정락 → 최팀장   GRANT   회계 approve  영구
```
- 컬럼: 일시 / 행위자 / 대상 / action(GRANT/DELEGATE/REVOKE/EXPIRE) / 권한 / 비고.
- `permission_audit_log`는 **append-only**, 행 우측 자물쇠 아이콘.

## 10. 01 코드팀 이식 메모

1. **백엔드 미들웨어가 권한 평가의 단일 진실** — UI 노출 제어만으로 우회 가능 (R3 §6).
2. `permission_audit_log` 테이블은 1차부터 반드시 구현 (R3 §6 — 후추가 시 과거 복원 불가).
3. 1차 MVP: 4역할(OWNER/MANAGER/MEMBER/VIEWER) + 자원별 read/write/approve 3액션만 (R3 §6).
4. 위임 기능은 2차 페이즈 — 1차는 OWNER 직접 역할 배정만 활성화 권고 (R3 §6, 84명 규모 충분).
5. 단, 1차에도 위임 화면 라우트는 두되 비활성 토글 노출(향후 활성화 위치 고정).
6. `canUserDoAction()` 함수는 R3 §3 의사코드 그대로 구현, DENY 우선 단위 테스트 5건 이상.
