```

```

   - --topbar-h: clamp(84-108px)

## 4-A. 01 응답서 사후 검증 의무 (v3 — 2026-04-28 대표 승인 16:30)

### 4-A-1. 응답서 액면 수용 금지 (절대준수)
01 응답서 받을 때마다 빅터(05) 의무 절차:
1. **응답서의 grep 결과 수용 보류** — 표면 수치만 보지 말고
2. **응답서 외 추가 grep 의무 5종**:
   - 페이지별 옛 패턴 (예: sage-) 잔존 카운트 (각 핵심 페이지 0건 확인)
   - 페이지별 인라인 #hex 잔존 카운트 (각 페이지 0건 확인)
   - 독립 페이지 (login·guide 등 base.html 미상속) 별도 검증
   - 사이드바·도크·푸터 등 base 내부 컴포넌트 별도 검증
   - 모바일 반응형 페이지 (1100px 이하) 별도 검증
3. **차이 발견 시 즉시 보강 발주서 발행** — 대표 점검 전에

### 4-A-2. "전체 시스템 적용" 정의
- = base.html 상속 페이지 (~120) + 독립 페이지 (login·guide·print 등)
- 발주서에 **명시적 파일 리스트** 포함 의무
- 카테고리별 처리 방식 명시 (alias / 정밀 정리 / 재작성)

### 4-A-3. 발주서 §grep 표준 5 카테고리 (다음 발주부터 적용)
| 카테고리 | 검증 종류 | PASS 기준 | 항목 수 |
|---|---|---|---|
| **A. 신규 패턴 추가** | 새 토큰·클래스·함수 추가 | ≥1 | 5~6 |
| **B. 옛 패턴 잔존** | 폐기되어야 할 변수·클래스·색상 | 0 | 5~6 |
| **C. 페이지별 적용** | 핵심 N개 페이지의 옛 잔존 | 각 페이지 0 | 5~6 |
| **D. 인라인 색상 정리** | `style="*#hex"` 직접 색상 | 0 | 5~6 |
| **E. 페르소나 walkthrough** | 시각 검증 시나리오 | 정성 | 6~12 |

→ 카테고리 5개 × 항목 5~6 = **30종 grep + walkthrough 의무**

## 4. 결함 발견 시 처리 분기 (v2 — 2026-04-28 대표 직접 지시 반영)

> **신 프로토콜 핵심**: *"변경한 디자인을 내가 먼저 확인하고 실무팀에 지시할께."*
> 디자인 변경은 무조건 **빅터 시안 → 대표 검토 → 승인 → 01 지시서** 순서 (직접 핫패치 금지).
> 단순 결함 수정(라벨·grep 0건 회복 등)은 핫패치 가능, 디자인 변경은 시안 우선.

### 4-1. 단순 결함 수정 (라벨·grep·옛 패턴 잔존 등 — 시각 변화 없음)
```
빅터(05) 즉시 핫패치
  ↓
대표 보고 (요약 + 검증 grep PASS)
  ↓
_TO_01 인계 문서 발행 (01이 BAT 갱신 + 인지)
```

### 4-2. 디자인 변경 (시각·레이아웃·구조 변경 — 작든 크든)
```
빅터(05) 디자인 시안 작성
  - 03_시안/ 폴더에 PREVIEW HTML 파일
  - 명시적 PREVIEW 배너 (대표 검토 대기 표시)
  - _TO_01 지시서는 _DRAFT_대표승인대기/ 폴더에 보관 (HOLD 노트)
  ↓
대표 보고: "프리뷰 X 위치, 검토 부탁"
  ↓
대표 검토 (파일 직접 열어 시각 확인)
  ↓
[승인] 빅터 → _TO_01 지시서 _세션01_전달/ 으로 이동 + 01 작업 시작
       → 01 작업 → 04 회귀 → 빅터 grep PASS → 정상 사이클
[거절·수정] 빅터 프리뷰 재작성 → 다시 검토
```

### 4-3. 페르소나 영향 핵심 결함 (사용성 마비)
```
빅터(05) _DIAG 진단서 + _TO_04 회귀 의뢰 + _TO_09 보고 (3종 즉시 발행)
  ↓
디자인 변경 동반 시 → §4-2 시안 우선 적용
디자인 변경 없는 경우 (라벨·grep) → §4-1 즉시 핫패치
  ↓
긴급 우선순위 처리
```
