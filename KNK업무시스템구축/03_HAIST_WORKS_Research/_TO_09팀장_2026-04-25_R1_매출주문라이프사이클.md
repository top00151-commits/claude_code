# R1 - 한국 ERP 매출 주문 라이프사이클 표준
> 발신: 03 Research / 수신: 09 팀장 + 04 운영테스트
> 일자: 2026-04-25 / 분량: 약 130줄

## 1. 핵심 결론 (3줄)
- 한국 중소제조업(40~100명) 표준 매출 사이클은 **견적(QT) → 수주(SO) → 생산지시(WO) → 출하(DO) → 세금계산서/매출인식(INV) → 수금(RC)** 의 6단계가 사실상 산업 표준이며, 더존·영림원·SAP B1 모두 동일 골격을 사용한다.
- 매출 인식 시점은 **K-IFRS 1115호(고객과의 계약에서 생기는 수익)** 의 "통제 이전 시점" 원칙에 따라 일반 제조업은 **출하 완료(인도) 시점** 또는 **검수 완료 시점** 중 계약 조건에 따라 결정된다.
- 주문 상태값(enum)은 ERP별로 명칭이 다르지만 의미상 7~9개로 수렴하며, KNK는 **DRAFT/CONFIRMED/IN_PRODUCTION/READY_TO_SHIP/SHIPPED/INVOICED/PAID/CANCELLED** 8개 상태로 표준화 권고.

## 2. 표준 흐름 (도식)

```
[고객 RFQ]
   │
   ▼
①견적 QT ─────► (반려/재견적)
   │ 승인
   ▼
②수주 SO (계약확정) ─────► (취소/변경)
   │ 자재 가용성 체크
   ▼
③생산지시 WO ─────► (지연/품질 NG → 재작업)
   │ 완성
   ▼
④출하 DO (납품지시 + 거래명세서)
   │ 인도/검수
   ▼
⑤매출인식 INV (세금계산서 발행 · 국세청 e-Tax)
   │ 여신·결제조건
   ▼
⑥수금 RC (입금확인 · 채권 마감)
```

| 단계 | 한국어 통칭 | 표준 도큐먼트 | 시스템 액션 |
|---|---|---|---|
| ① | 견적서 | Quotation | 가격·납기·결제조건 제시 |
| ② | 주문서/수주 | Sales Order | 재고할당·생산투입 |
| ③ | 작업지시서 | Work Order | BOM 폭발·자재출고 |
| ④ | 거래명세서 | Delivery Note | 출고처리·재고차감 |
| ⑤ | 세금계산서 | Tax Invoice | 매출인식·채권발생 |
| ⑥ | 입금표 | Receipt | 채권상계·정산 |

## 3. 데이터 모델 권고 (KNK 적용 시)

| 테이블 | 핵심 필드 | 비고 |
|---|---|---|
| `sales_orders` | id, customer_id, status, total_amount, due_date, created_by | 상태 enum 8종 |
| `sales_order_items` | so_id, product_id, qty, unit_price, allocated_qty | 재고할당 추적 |
| `quotations` | id, customer_id, valid_until, version | 버전관리 필수 |
| `work_orders` | id, so_id, bom_id, planned_start, actual_end | 생산진행률 |
| `delivery_notes` | id, so_id, shipped_at, carrier, tracking_no | 출하증빙 |
| `invoices` | id, so_id, tax_invoice_no, issued_at, due_date | 국세청 e-Tax 연동키 |
| `receipts` | id, invoice_id, received_at, amount, method | 부분수금 지원 |
| `order_status_history` | so_id, from_status, to_status, changed_by, changed_at | 감사로그 |

상태 enum (PostgreSQL 기준):
```sql
CREATE TYPE sales_order_status AS ENUM (
  'DRAFT', 'CONFIRMED', 'IN_PRODUCTION', 'READY_TO_SHIP',
  'SHIPPED', 'INVOICED', 'PAID', 'CANCELLED'
);
```

## 4. UI 시사점 (05 디자인팀이 받을 것)

1. **상태 배지(Status Pill)** — 8가지 색상 토큰 분리. CANCELLED는 회색, PAID는 녹색, IN_PRODUCTION은 주황 권장.
2. **타임라인 뷰** — 주문 상세 화면 우측에 ①~⑥ 스텝퍼(Stepper) 표시. 현재 단계 강조 + 예정일·실제일 동시 노출.
3. **가용 액션 버튼은 상태별로 다름** — CONFIRMED 상태에서만 "생산지시 발행", SHIPPED 상태에서만 "세금계산서 발행" 노출 (역행 방지).
4. **부분 출하(Partial Delivery) 처리** — 1개 SO가 N개 DO로 쪼개지는 케이스가 한국 중소제조에서 빈번. 진행률 % 표시 필수.
5. **세금계산서 발행 버튼은 별도 권한** — 회계담당자만 보여야 함 (R3 권한 위임과 연계).

## 6. KNK 적용 권고 (4-5줄)
- 8단계 enum을 그대로 채택하되, 초기 MVP는 DRAFT/CONFIRMED/SHIPPED/PAID 4단계로 단순화 후 점진 확장.
- 세금계산서 발행은 자체 구현 대신 **팝빌·바로빌 API 연동(BPO)** 으로 위임 (외부 연결 사전 승인 필요).
- 매출 인식 시점은 KNK 사업특성상 "출하 완료" 시점 권고 — 검수 조건부 계약은 별도 플래그로 처리.
- 상태 전이 검증은 백엔드 finite state machine으로 강제 (UI만으로 막으면 우회 가능).
- 부분 출하/부분 수금은 1:N 관계로 설계 — 1:1로 만들면 6개월 내 재설계 필수.
