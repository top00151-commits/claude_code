### B-2. FTS5 vs LIKE 검토

**현 LIKE 쿼리 부담 평가**:
- 카테고리 9개 × 평균 4 LIKE = 36 LIKE 펀치 / 매 검색
- 행수 추정: parts ~수천 / orders ~수백 / customers ~수십 — 현 시드 84명 + 시범 데이터 규모에서 **LIKE 인덱스 없이도 100ms 이하** (정찰 추정)
- 정직성: 실측 미실시 — 04 운영검증에 부하 측정 위임 권고

**FTS5 도입 검토**:
- 가능: SQLite 빌트인 모듈 — 외부 자산 X (정책 통과)
- 비용: virtual table 신설 9개 + INSERT/UPDATE/DELETE 트리거 9개 = **테이블 신설 9건** → 핫패치 G1~G5 보존 정책 안전 영역이긴 하나 스키마 비대화
- 결론: **현 시점 불필요**. 데이터 규모 1만건 돌파 시 재검토 권고.

**권고**: LIKE 유지 + 부분 인덱스 (`CREATE INDEX IF NOT EXISTS`) 추가만으로 충분.
