# R3 - 권한 위임 UX 패턴 (CEO → 팀장 → 팀원 3계층)
> 발신: 03 Research / 수신: 09 팀장 + 04 운영테스트
> 일자: 2026-04-25 / 분량: 약 140줄

## 1. 핵심 결론 (3줄)
- B2B SaaS 권한 위임 UX는 **(1) Role Matrix, (2) Per-Resource Share, (3) Group Inheritance, (4) Delegation Token, (5) Approval Workflow** 5가지 패턴이 시장 표준이며, KNK 시나리오(대표→팀장→팀원 재위임)는 **(3) + (4) 조합** 이 적합하다.
- 권한 충돌 시 **"가장 제한적인 권한이 우선(Deny overrides Allow)"** 원칙이 GitHub·Atlassian·AWS IAM 공통 표준 — KNK도 이 규칙 채택 권고.
- 권한 위임은 반드시 **(a) 위임자 본인의 권한 범위 내에서만, (b) 만료일 설정 가능, (c) 위임 이력은 immutable 로그** 3원칙을 만족해야 한다.

## 2. 표준 흐름 (5가지 UI 패턴 비교)

| 패턴 | 대표 사례 | 장점 | 단점 |
|---|---|---|---|
| ① Role Matrix | Notion, Slack | 단순, 학습용이 | 세분화 한계 |
| ② Per-Resource Share | Google Drive, Notion 페이지 | 직관적 | 대규모 시 관리지옥 |
| ③ Group Inheritance | Atlassian, Azure AD | 확장성↑, 조직도 매핑 | 상속 추적 복잡 |
| ④ Delegation Token | AWS IAM AssumeRole, OAuth scope | 한시적 위임, 만료자동 | UI 어려움 |
| ⑤ Approval Workflow | ServiceNow, Jira Service Mgmt | 감사대응 강함 | 속도 느림 |

KNK 시나리오 매핑:
```
[CEO 김정락]                              ← 모든 권한 보유 (Owner)
   │
   │ ① 위임 (Delegation): "물류권한"을 팀장에게 부여
   ▼
[팀장 (예: 자재팀장)]                      ← Manager 그룹 (③ Inheritance)
   │
   │ ② 재위임: "입고처리"만 팀원에게 부여 (만료 1개월)
   ▼
[팀원 A·B·C]                              ← Member 그룹 + 한시 토큰 (④)
```

권한 충돌 해결 규칙 (Deny overrides):
```
1. 명시적 거부(DENY) > 명시적 허용(ALLOW) > 기본값(상속)
2. 그룹 권한 합집합 적용. 단 DENY는 항상 우선.
3. 위임된 권한은 위임자의 권한을 초과할 수 없다 (No escalation).
4. 위임자의 권한이 회수되면 하위 위임도 자동 무효화 (Cascade revoke).
```

## 3. 데이터 모델 권고 (KNK 적용 시)

| 테이블 | 핵심 필드 | 비고 |
|---|---|---|
| `users` | id, email, name, status | 직원 마스터 |
| `roles` | id, code (OWNER/MANAGER/MEMBER/VIEWER), name | 시스템 정의 |
| `permissions` | id, resource (sales/stock/hr/...), action (read/write/approve) | 원자권한 |
| `role_permissions` | role_id, permission_id | 역할-권한 매핑 |
| `user_roles` | user_id, role_id, scope (전사/팀/특정자원), granted_by, granted_at, expires_at | 핵심 |
| `delegations` | id, from_user_id, to_user_id, permission_id, scope, expires_at, revoked_at | 위임 추적 |
| `permission_audit_log` | id, actor_id, target_user_id, action (GRANT/REVOKE/DELEGATE), at, reason | append-only |

핵심 SQL enum:
```sql
CREATE TYPE perm_effect AS ENUM ('ALLOW', 'DENY');
CREATE TYPE delegation_status AS ENUM ('ACTIVE', 'EXPIRED', 'REVOKED');
```

권한 평가 의사코드:
```
function canUserDoAction(user, resource, action):
  perms = collectAllPerms(user)        # 직접 + 그룹상속 + 위임 합집합
  if any(p.effect == DENY for p in perms): return false
  if any(p.effect == ALLOW for p in perms): return true
  return false  # 기본 거부
```

## 4. UI 시사점 (05 디자인팀이 받을 것)

1. **권한 위임 화면은 "보내기/회수" 두 탭** — Notion 공유 모달 패턴 차용. 좌측 "내가 위임한 권한", 우측 "내가 받은 권한".
2. **위임 모달 필수 필드 4개**:
   - 누구에게 (사용자 검색)
   - 무엇을 (권한 체크박스, 본인 보유 권한만 노출)
   - 언제까지 (만료일, 기본 30일 / 영구)
   - 사유 (선택, 감사로그용)
3. **권한 충돌 시 사용자 피드백** — "이 작업은 OOO 권한 부족" 토스트 + "팀장에게 요청" 버튼 (Atlassian 패턴).
4. **이력 화면(Audit Trail)** — 시간역순 타임라인. "2026-04-25 14:30 김정락 → 박팀장 에게 '물류 입고처리' 권한 부여 (만료 2026-05-25)".
5. **재위임 가능 토글** — "팀장이 팀원에게 재위임 가능" 옵션. OFF가 기본값 (안전한 기본값).
6. **회수(Revoke) 버튼은 빨간색 + 2단계 확인** — Cascade 영향 미리 보여주기: "이 권한을 회수하면 박팀장이 김사원에게 위임한 권한 2건도 함께 무효화됩니다."

## 6. KNK 적용 권고 (4-5줄)
- 1차 MVP: **4개 역할(OWNER/MANAGER/MEMBER/VIEWER) + 자원별 read/write/approve 3액션** 으로 단순 시작.
- 위임 기능은 2차로 미루고, 1차는 OWNER(대표)가 직접 역할 배정만 가능하게 — 이것만으로 84명 규모는 충분.
- 위임이 꼭 필요하다면 **만료일 필수 + 재위임 OFF 기본값** — 권한 폭주 방지.
- `permission_audit_log`는 1차부터 반드시 구현. 나중에 추가하면 과거 데이터 복원 불가.
- "Deny overrides Allow" 규칙은 백엔드 미들웨어에서 강제. UI 노출 제어만으로는 우회 가능.
